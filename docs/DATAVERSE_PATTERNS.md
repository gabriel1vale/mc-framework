# Dataverse Patterns

Gotchas and essential patterns when working with Dataverse via the SDK or direct API.

## FormattedValue for lookups, choices, dates

### Problem

Types autogenerated by `pac code add-data-source` declare flat properties like `crd1_brandname?: string`, but the SDK does not populate that field. What exists at runtime is the OData annotation:

```
_<schemaname>_value@OData.Community.Display.V1.FormattedValue
```

Trusting the type leads to a UI showing all `—` or `undefined`.

### Solution

Helper:

```typescript
const FORMATTED = '@OData.Community.Display.V1.FormattedValue';

export function fmt(record: object, field: string): string {
  const value = (record as Record<string, unknown>)[`${field}${FORMATTED}`];
  return typeof value === 'string' && value.length > 0 ? value : '—';
}
```

Usage:

```typescript
fmt(orderline, '_crd1_product_value');     // product name
fmt(orderline, '_crd1_region_value');      // region name
fmt(orderline, 'crd1_type');               // choice label
fmt(orderline, 'createdon');               // formatted date
```

### For writes

To SET a lookup, use OData bind syntax (not the flat name):

```typescript
const payload = {
  // CORRECT:
  'crd1_Product@odata.bind': `/crd1_products(${productId})`,
  'crd1_Region@odata.bind': `/crd1_regions(${regionId})`,

  // WRONG (will be ignored):
  crd1_product: { id: productId },
  crd1_productname: 'My Product',
};

await Crd1_orderlinesService.create(payload as never);
```

Note that the **schema name is case-sensitive** in `@odata.bind` (e.g. `crd1_Product` with capital P, not `crd1_product`).

### To clear a lookup

```typescript
await Service.update(id, {
  'crd1_Product@odata.bind': null
} as never);
```

## Lookup tracking via lookup, not string

### Problem

Some tables have a lookup field (e.g. `crd1_region` pointing to `crd1_regions`) AND an equivalent string field (e.g. `crd1_regionname`). It is tempting to populate only the string and ignore the lookup. **Don't.**

The Dataverse **rollup engine** marks the parent record as dirty via the lookup relationship. Records created without populated lookup (even with the correct equivalent string) are **invisible to the rollup tracker** — and `CalculateRollupField` returns the cached value.

### Solution

Always populate the lookup:

```typescript
const payload = {
  'crd1_Product@odata.bind': `/crd1_products(${productId})`,
  'crd1_Region@odata.bind': `/crd1_regions(${regionId})`,  // ESSENTIAL for rollup tracking
  crd1_quantity: 5,
  crd1_type: 100000000,
};
```

The string `crd1_regionname` can also be populated (if the schema requires) but **never instead of** the lookup.

## Autonumber columns: pass explicit value in bulk imports

### Problem

Some custom tables have a string column with a plugin that generates autonumber. When you create records via SDK/MCP/Web API without passing that column, the plugin kicks in and generates **short random values** (e.g. `BJ`, `RY`, `M`) instead of sequential numbers.

If your app expects `crd1_seqid` to be sequential numeric (e.g. for ordered display, or for `getNextId()` lookups), this breaks.

### Solution

For bulk imports: discover the last value + 1, and pass explicitly:

```typescript
async function getNextId(): Promise<string> {
  let max = 0;
  let token: string | undefined;
  do {
    const result = await Service.getAll({
      select: ['crd1_seqid'],
      maxPageSize: 5000,
      skipToken: token,
    });
    if (!result.success) throw new Error(result.error?.message);
    for (const r of result.data) {
      const n = parseInt(r.crd1_seqid ?? '', 10);
      if (!Number.isNaN(n) && n > max) max = n;
    }
    token = result.skipToken;
  } while (token);
  return String(max + 1);
}

// Bulk:
const startId = parseInt(await getNextId(), 10);
const rowsWithIds = rows.map((r, i) => ({
  ...r,
  crd1_seqid: String(startId + i),
}));
```

Caveat: race condition if multiple processes import simultaneously. If that is a real risk, use a lock or single-threaded import.

## `pac code add-data-source` for Dataverse

To add a Dataverse-native table to a Code App:

```bash
pac code add-data-source -a dataverse -t <table-logical-name>
```

`-a dataverse` is the literal "dataverse" — **not** `/providers/Microsoft.PowerApps/apis/shared_commondataserviceforapps` (that is connector mode, requires `--connectionId`).

Code Apps support two models:
- **Dataverse direct binding** (recommended) — no connectionId, ambient user auth
- **Connector-based** — requires connectionId; useful for SharePoint, SQL, Teams, etc.

For Dataverse, always the first. Generates `<Table>Service.ts` + `<Table>Model.ts` in `src/generated/`.

## Custom API / Action invocation at runtime

To call a custom action or function (e.g. `CalculateRollupField`, `WhoAmI`, custom plugin actions), register the API at runtime in `dataSourcesInfo` before any call:

```typescript
import { dataSourcesInfo } from '../../.power/schemas/appschemas/dataSourcesInfo';

const calculateRollupFieldApi = {
  path: "CalculateRollupField(Target=@target,FieldName='{fieldName}')?@target={target}",
  method: 'GET',
  parameters: [
    { name: 'target', in: 'path', required: true, type: 'string' },
    { name: 'fieldName', in: 'path', required: true, type: 'string' },
  ],
};

(() => {
  const dv = dataSourcesInfo as unknown as Record<string, { apis?: Record<string, unknown> }>;
  if (!dv.crd1_products) {
    console.warn('[rollup] crd1_products data source not found');
    return;
  }
  if (!dv.crd1_products.apis) dv.crd1_products.apis = {};
  dv.crd1_products.apis.CalculateRollupField = calculateRollupFieldApi;
})();
```

**Why direct mutation**: `PowerDataSourcesInfoProvider` is a singleton and only reads `dataSourcesInfo` on the first `getClient` call. Subsequent copies are discarded. Mutating the original object (which the singleton holds by reference) propagates.

Then invoke:

```typescript
const result = await client.executeAsync({
  dataverseRequest: {
    action: 'customapi',
    parameters: {
      operationName: 'CalculateRollupField',
      tableName: 'crd1_products',
      body: {
        target: JSON.stringify({ '@odata.id': `crd1_products(${productId})` }),
        fieldName: 'crd1_stock<X>',
      },
    },
  },
});
```

## Code Apps are sandboxed — no direct fetch

Code Apps **cannot** make `fetch`/`XMLHttpRequest` directly to external APIs at runtime (sandbox restriction). All communication must go through:

- `@microsoft/power-apps` SDK (Dataverse direct binding)
- Connectors registered via `pac code add-data-source`
- Custom APIs / Functions invoked via the pattern above

If your app needs an external API: wrap it in a connector, or proxy via a Dataverse plugin.

For **dev-time/AI-driven** ops (not app runtime), use `az` + Web API direct (see `BULK_OPS_PATTERNS.md`).

## Choices (option sets) with numeric values

Choices in Dataverse are int values, not strings. E.g. `crd1_type` = `100000000` (Inbound) or `100000001` (Outbound).

```typescript
const TYPE_INBOUND = 100000000;
const TYPE_OUTBOUND = 100000001;

// On create:
{ crd1_type: TYPE_INBOUND }

// On filter:
filter: `crd1_type eq ${TYPE_INBOUND}`

// On comparison:
if (Number(record.crd1_type) === TYPE_INBOUND) { ... }
```

Values are defined in maker (`make.powerapps.com`). Verify via `describe_table` or `read_query` to confirm.

To show the label: `fmt(record, 'crd1_type')` returns `"Inbound"` / `"Outbound"` (FormattedValue).

## Boolean fields

In Dataverse, boolean is represented as `true`/`false`. FormattedValue returns `"Yes"`/`"No"` according to locale.

```typescript
{ crd1_isadjustment: true }       // create with adjustment-mode = true
filter: `crd1_isadjustment eq true`
filter: `crd1_isadjustment ne false`   // captures true OR null
```

Watch out for legacy records (created before the column existed) that end up with `null` — different from `false`.

## Currency / decimal precision

Currency and decimal columns return as `number` at runtime. Beware of precision:

```typescript
const total = (record.crd1_amount ?? 0).toFixed(2);
// vs
const total = Math.abs(Number(record.crd1_amount ?? 0));
```

For totals, sum and round at the end (not at each step).

## Date / DateTime

Dates return as ISO 8601 strings. For display: `fmt(record, 'createdon')` applies the user locale.

For filters:

```typescript
filter: `createdon ge 2024-01-01T00:00:00Z`
filter: `createdon le 2024-12-31T23:59:59Z`
```

Always UTC in filters.

## Soft delete vs hard delete

By default, `service.delete(id)` does **hard delete** (record disappears). If the table has soft delete configured, it is an update setting `statecode=Inactive, statuscode=Inactive`.

Check via `describe_table` if the table has a `statecode` column. If yes, you can soft-delete:

```typescript
await Service.update(id, {
  statecode: 1,   // Inactive
  statuscode: 2,  // Inactive (specific reason code varies)
} as never);
```
